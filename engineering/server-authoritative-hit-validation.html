Engineering Case Study

Server-Authoritative Hit Validation

Combat pipeline that preserves responsive local feedback while making the server the final authority for damage and hit legality.

Language: Lua | Runtime: Roblox | Focus: Networking and Fairness

Overview

This validation model separates player feel from game truth. Clients render immediate shot feedback for responsiveness, while the server independently validates each reported hit before applying damage.

Problem

Fully client-trusted hit registration is vulnerable to manipulation and desync. Fully server-simulated shooting can feel laggy. The challenge is balancing fairness and responsiveness under real-world latency.

Design Goals

  • Server decides final hit validity and damage.
  • Client still receives immediate audiovisual shot feedback.
  • Validation tolerates normal latency without rewarding impossible shots.
  • System remains debuggable with clear rejection reasons.

Architecture

Client Fires Weapon Event
Client Predicts Effects Locally
Server Reconstructs Shot State
Raycast and Rule Validation
Apply Damage or Reject Event

Implementation

Each shot event carries compact context: origin, direction, timestamp, and weapon identifier. On receipt, the server checks fire-rate windows, validates shooter state, reconstructs a raycast path, and only then commits damage.

Important boundary: the client can request a hit, but cannot declare a hit.

Engineering Challenges

Latency and fairness

Validation windows were tuned to avoid rejecting normal latency cases while still catching physically impossible claims.

Rate-of-fire exploits

Server enforces weapon cadence with timestamp checks, preventing client-side fire-speed manipulation.

Debuggability

Reject paths emit categorized reasons, making it possible to tune thresholds from telemetry instead of guesswork.

Code Examples

Server Shot Validation Skeleton

local function validateShot(player, payload)
    if not isFireRateLegal(player, payload.weaponId, payload.timestamp) then
        return false, "fire_rate"
    end

    if not isStateLegal(player, payload) then
        return false, "player_state"
    end

    local rayResult = serverRaycast(payload.origin, payload.direction, payload.weaponId)
    if not rayResult.hit then
        return false, "no_hit"
    end

    if not isTargetLegal(player, rayResult.instance) then
        return false, "target_invalid"
    end

    return true, rayResult
end

Applying Damage After Validation

local ok, result = validateShot(player, payload)
if not ok then
    logRejectedShot(player, result)
    return
end

applyWeaponDamage(player, result.instance, payload.weaponId)
replicateConfirmedHit(player, result)

Tradeoffs

  • Strict validation improves fairness but can increase false negatives if thresholds are too tight.
  • Client prediction improves feel but requires reconciliation paths for rejected events.
  • More validation layers improve security but add computational overhead per shot.

Future Improvements

  • Add lag-compensated rewind for target position validation.
  • Build dashboards for rejection reason distribution over time.
  • Attach confidence scoring to suspicious shot clusters.

Why I Am Proud Of It

This system let me keep gunplay responsive without giving up authority. It turned networking fairness from an afterthought into a first-class design constraint.

Lessons Learned

  • Authority boundaries should be explicit and documented from the start.
  • Reject telemetry is as important as accepted-hit telemetry.
  • Security and player feel must be designed together, not in sequence.
