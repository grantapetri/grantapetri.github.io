Engineering Case Study

Dynamic Weapon Sway

A procedural first-person weapon motion system driven by spring simulation instead of hand-authored animation clips.

Language: Lua | Runtime: Roblox | System: Procedural Animation

Overview

This system computes weapon sway every frame from camera movement and character velocity. The output is physically plausible motion that scales naturally with player input, while exposing tuning knobs for gameplay feel.

Problem

Animation-only weapon sway produced repetition and required clip authoring for each weapon variant. I needed a unified runtime approach that could support many weapons without multiplying animation maintenance cost.

Design Goals

  • Responsive feel tied directly to camera and movement deltas.
  • Low per-weapon setup cost through parameterized configuration.
  • Stable behavior at high and low frame rates.
  • Minimal visual jitter while preserving readable motion.

Architecture

Input Sampling: Camera Delta + Velocity
Target Sway Computation
Spring Integrator Update
Transform Blend and Clamp
Apply To Weapon Viewmodel

Implementation

The update loop computes a target offset from recent camera rotation and movement, then pushes that target through damped springs for rotational and positional channels. Final offsets are clamped and blended before being applied to the viewmodel transform.

Core implementation choice: use springs as the single source of temporal smoothing rather than stacking ad-hoc lerps.

Engineering Challenges

Frame-rate consistency

Spring integration was tuned around delta time to avoid changing weapon feel across different hardware.

Input noise and jitter

Raw camera deltas can spike. I normalized and bounded signal ranges before feeding the springs.

Per-weapon tuning burden

I created grouped presets so weapons inherit baseline feel and only override where necessary.

Code Examples

Target Sway Derivation

local camDelta = currentCameraCFrame:ToObjectSpace(lastCameraCFrame)
local rx, ry, _ = camDelta:ToOrientation()

local targetRot = Vector3.new(-rx, -ry, 0) * swayConfig.rotationScale
local targetPos = humanoidRoot.AssemblyLinearVelocity * swayConfig.positionScale

rotationSpring:Shove(targetRot)
positionSpring:Shove(targetPos)

Spring Output Application

local rot = rotationSpring:Update(dt)
local pos = positionSpring:Update(dt)

rot = Vector3.new(
    math.clamp(rot.X, -maxRot, maxRot),
    math.clamp(rot.Y, -maxRot, maxRot),
    0
)

local swayCF = CFrame.new(pos) * CFrame.Angles(rot.X, rot.Y, rot.Z)
viewmodel:SetPrimaryPartCFrame(camera.CFrame * swayCF)

Tradeoffs

  • Procedural sway is adaptive and cheap to scale, but requires careful parameter tuning to avoid floaty behavior.
  • Animation pipelines give direct artistic control, but increase maintenance as weapon count grows.
  • Spring-heavy systems reduce clip dependency, but demand disciplined constraints and clamping.

Future Improvements

  • Add context-aware sway profiles for sprinting, aiming, and low-stamina states.
  • Expose tuning sliders through in-game debug UI for rapid balancing.
  • Blend recoil and sway into a shared impulse framework.

Why I Am Proud Of It

This system improved feel immediately while removing repetitive animation setup work. It became a reusable building block instead of a one-off effect.

Lessons Learned

  • Motion quality comes from stable signal processing before stylization.
  • Reusable tuning surfaces matter as much as the underlying math.
  • Procedural animation can be production-friendly when constraints are explicit.
