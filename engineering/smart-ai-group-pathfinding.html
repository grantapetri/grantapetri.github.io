Engineering Case Study

Smart AI Group Pathfinding

Group-aware pursuit system that combines navigation paths with tactical weighting so AI enemies pressure players as coordinated units rather than queueing bots.

Language: Lua | Runtime: Roblox | Focus: Navigation and Group Behaviors

Overview

The system blends baseline pathfinding with role selection and spacing logic. Instead of every AI choosing the shortest route to the exact player position, enemies evaluate candidate approach vectors and choose complementary positions.

Problem

Naive pathfinding created predictable clumping and doorway traffic jams, especially in procedural maps. This reduced threat quality and made combat feel like a single-file queue rather than coordinated pressure.

Design Goals

  • Distribute enemies around the player instead of stacking at one point.
  • Prevent self-blocking in narrow spaces and dynamic geometry.
  • Retain deterministic fallback behavior when role assignment fails.
  • Keep runtime cost bounded for multiplayer sessions.

Architecture

Perception Update: Player + World State
Candidate Slot Generation Around Target
Score Slots: Distance, Risk, Congestion
Assign Group Roles and Destinations
Path Compute + Local Avoidance
Re-evaluate On Timer or Disturbance

Implementation

Every AI computes a set of candidate target offsets around the player. Offsets are scored by travel cost, angle diversity, and local crowding. A lightweight assignment pass picks destinations that maximize spread. Agents then path toward assigned slots and locally avoid teammates.

Key behavior shift: move from "chase the player position" to "occupy pressure slots around the player."

Engineering Challenges

Dynamic maps and blocked routes

Procedural layouts can invalidate assumptions quickly. The system uses periodic reassignment and fallback slots when paths collapse.

Group oscillation

Without hysteresis, agents swap roles too often. I introduced score stickiness and cooldowns to stabilize assignments.

Performance constraints

Full recomputation each frame is too expensive. I run staged updates and only recalc expensive scores on cadence or disturbances.

Code Examples

Slot Scoring Pass

local function scoreSlot(agent, slot, context)
    local pathCost = estimatePathCost(agent.position, slot.position)
    local crowdPenalty = getCrowdPenalty(slot, context.assignedSlots)
    local angleBonus = getAngleDiversityBonus(slot, context.playerForward)

    return (-pathCost * weights.path)
         + (angleBonus * weights.angle)
         - (crowdPenalty * weights.crowd)
end

Assignment Loop

for _, agent in ipairs(groupAgents) do
    local bestSlot, bestScore = nil, -math.huge

    for _, slot in ipairs(candidateSlots) do
        if not isReserved(slot) then
            local score = scoreSlot(agent, slot, context)
            if score > bestScore then
                bestScore = score
                bestSlot = slot
            end
        end
    end

    reserveSlot(bestSlot, agent)
    setAgentDestination(agent, bestSlot.position)
end

Tradeoffs

  • Coordinated behaviors feel better but require more orchestration than independent chasing.
  • Assignment stability improves readability but can delay reaction to sudden player movement.
  • Frequent re-scoring improves adaptation but increases CPU load.

Future Improvements

  • Add behavior archetypes so different enemy classes value slots differently.
  • Integrate line-of-sight confidence into slot scoring.
  • Introduce cooperative flanking triggers tied to sound events.

Why I Am Proud Of It

The system transformed encounter quality. Players report pressure from directionality and coordination, not just enemy count, which is exactly the design target.

Lessons Learned

  • Good AI pressure comes from spatial roles, not only pathfinding speed.
  • Small hysteresis rules can eliminate major behavior noise.
  • Procedural maps demand robust fallback logic in every AI layer.
