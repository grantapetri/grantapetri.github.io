Flagship Project

Site 23

Procedural horror built around a custom room-based generation framework. The project is less about random placement and more about solving constraints while preserving pacing and replayability.

Status: In Development | Engine: Unity | Genre: Procedural Horror

Overview

Site 23 is built as a procedural generation framework first and a game second. The generation begins from a start room and expands through doorway constraints. Every doorway becomes an expansion point, and each room placement must satisfy transform alignment and collision constraints before it can be accepted.

The practical goal is to produce maps that feel authored. Randomness decides options, but constraints decide what is legal. That separation has been the most important architectural decision in this project.

Generation Pipeline

Start Room
Collect Exposed Doorways
Select Doorway + Shuffle Blueprints
Find Compatible Door + Solve Transform
Collision Validation
Place Room + Queue New Doors
Depth Reached -> Seal Dead Ends

Step 1

Select a doorway from the queue and shuffle candidate room blueprints.

Step 2

Match compatible door metadata and solve rotation + translation so doors align and face opposite directions.

Step 3

Reject overlaps against existing geometry. Accept only legal placements and append exposed doors to the queue.

Two Generation Phases

Root Generation

Builds a guaranteed critical path so progression can always complete. This phase prioritizes structural reliability over variation.

Expansion Generation

Branches outward from the critical path to create optional routes, side spaces, and replayability without breaking progression.

Engineering Challenges

Transform Alignment for Arbitrary Rooms

Door alignment became a rigid transform problem: room prefabs arrive in arbitrary orientation, so I solve a transform that rotates and translates the candidate room until the chosen door pair is co-located and facing opposite normals.

Collision Validation at Scale

As prefab count grew, naive overlap checks became expensive. I moved validation earlier in the pipeline and reject impossible candidates as soon as doorway constraints fail.

Natural Doorway Detection

Some doorways connect naturally after independent placements. I detect these post-placement and convert them into valid connections to avoid blocking plausible paths.

Multi-Floor Support

Vertical expansion adds extra constraints: floor transitions, stair clearance, and vertical collision checks. This shifted the generator from random assembly toward constraint solving.

This project is fundamentally a constraint solver. Randomness proposes candidates. Constraints decide the map.

What I Am Most Proud Of

The generated layouts read as handcrafted despite being assembled at runtime. The framework supports large prefab sets, multiple floors, and naturally occurring room connections without hand-linking every route.

Lessons Learned

  • Separating progression from exploration produces more believable maps.
  • Procedural generation is about gameplay structure as much as randomness.
  • Constraint quality matters more than random seed quality.
